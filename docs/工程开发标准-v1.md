# 工程开发标准 v1

## 1. 文档目标

本文档用于统一 YinYuan 仓库当前阶段的开发事实基线、工程约束与执行口径，避免以下问题：

- 实现现状与导航文档脱节
- 仓库约定分散在多个文件中，执行时口径不一致
- 自动化工具、人工开发者、后续维护者对“以什么为准”理解不一致

本文档面向当前仓库整体开发，覆盖：

- 当前已实现能力的事实基线
- 仓库结构与目录边界
- 构建、运行、测试口径
- 工程约束与提交规范
- 文档更新与冲突处理规则

## 2. 生效范围

- 仓库根目录：`YinYuan/`
- Rust 主工作区：`y2m-rs/`
- 前端 monorepo：`frontend-monorepo/`
- 文档目录：`docs/`

## 3. 事实来源优先级

当多个文档描述不一致时，按以下优先级解释：

1. [`当前实现说明.md`](./当前实现说明.md)：当前已落地实现的**事实基线**
2. [`工作进度.md`](./工作进度.md)：当前阶段的任务状态与优先级总表
3. [`agent.md`](../agent.md)：全仓工程约束 + 仓库导航、运行入口、测试入口与开发提示（合并原 `CONVENTIONS.md` / `CLAUDE.md`；根目录两文件为重定向存根）
4. 其他需求、设计、API、运维类文档

解释规则：

- 涉及“代码现在已经实现了什么”，以 `当前实现说明.md` 为准
- 涉及“代码应该怎么写、如何提交、如何组织进程”“仓库在哪里、命令从哪里跑、测试怎么分类”，均以 **`agent.md`** 为准
- 若 **`agent.md`** 与 `当前实现说明.md` 冲突，必须以 `当前实现说明.md` 为准，并尽快同步修正 **`agent.md`**

## 4. 当前阶段标准结论

### 4.1 仓库结构标准

当前仓库按以下结构理解：

```text
YinYuan/
  docs/                ← 需求、设计、实现说明、进度、规范
  y2m-rs/              ← 当前 Rust 主工作区
  frontend-monorepo/   ← 前端 monorepo
  agent.md             ← 工程约定 + 仓库导航与开发提示
  CONVENTIONS.md       ← 重定向至 agent.md（兼容）
  CLAUDE.md            ← 重定向至 agent.md（兼容）
```

当前 Rust 代码主线在 `y2m-rs/`，所有 `cargo` 相关命令应从该目录执行，或显式带 `--manifest-path y2m-rs/Cargo.toml`。

### 4.2 当前实现标准

当前主实现以 `y2m-rs` 为准，已形成以下事实：

1. 协议版本为 `v3`
2. 系统已具备服务端、客户端核心库、CLI 三部分
3. 控制面走 JSON 文本帧，文件内容走二进制分片帧
4. 当前已实现消息类型包括：
   - `text`
   - `json`
   - `command`
   - `command_result`
   - `file_offer`
   - `file_accept`
   - `file_reject`
   - `file_complete`
   - `file_abort`
5. 当前服务端已支持：
   - `text` 单播与组内广播
   - `json` 单播与组内广播
   - `command` 单播与组内广播
   - `command_result` 单播与组内广播
   - `file_*` 控制消息单播与组内广播
   - 二进制 chunk 向已 `accept` 的接收端复制转发
6. 当前 CLI 已支持：
   - `init`
   - `run`
   - `send text/json/command/file`
   - `chat`
   - 文件接收确认、拒绝、中止与状态查看
   - 自动重连的最小可用实现
   - `command_result` 按 `request_id` 聚合输出
7. 当前文件传输已具备最小可用链路，但更正式的断线恢复与状态机仍在持续收敛

### 4.3 冲突修正规则

当前明确确认以下口径：

- 历史导航文档中“`command` 与 `file_*` 仅支持单播”的旧描述，不再代表当前真实实现（以 **`agent.md`** 现行表述为准）
- 当前实现以 [`当前实现说明.md`](./当前实现说明.md) §3.2、§3.4、§3.10 为准，即：
  - `command` 支持组内广播
  - `command_result` 支持组内广播
  - `file_*` 控制消息支持组内广播

后续若新增能力已经落地，必须先更新 `当前实现说明.md`，再同步导航类文档。

## 5. 代码组织标准

### 5.1 Rust 工作区标准

当前 `y2m-rs/` 结构按以下职责理解：

```text
y2m-rs/
  crates/common/       ← 协议模型、公共常量、二进制头、跨平台辅助
  crates/server/       ← WebSocket 服务端与路由主链路
  crates/client-core/  ← 连接、运行时、协议收发、插件分发
  src/                 ← y2m CLI 入口与命令实现
  tests/               ← 集成测试与进程级 e2e
```

职责边界：

1. `common` 只放协议与公共基础能力
2. `server` 负责连接、路由、会话、传输注册与服务端规则
3. `client-core` 负责传输与插件分发，不直接承担 UI
4. `src/` 负责 CLI 用户交互、命令解析与本地控制台行为
5. 文件状态统一经 `LocalFileStore` / `LocalFileTransfer` 管理，不允许额外散落并行状态

### 5.2 状态变更标准

涉及文件状态、本地控制台状态、重连回放时，遵循以下规则：

1. 所有文件状态变更必须走统一状态对象与迁移入口
2. 非法迁移使用结构化错误表达，不退化成布尔返回值
3. 新增本地状态时，必须接入重连清理路径
4. 新增状态机语义时，应优先扩展现有状态表，而不是新增平行布尔标志

### 5.3 跨平台命令执行标准

当前标准如下：

1. 协议层默认 shell helper 仍由 `y2m-common` 提供
2. Windows 协议辅助默认仍可返回 `cmd /C`
3. 但当前 CLI 在执行 `command` 事件时，Windows 实际使用 `pwsh`
4. 新增命令执行逻辑时，不得绕开现有跨平台辅助与现有执行约定

## 6. 构建、运行与测试标准

### 6.1 构建标准

Rust 构建命令从 `y2m-rs/` 执行：

```bash
cargo build
cargo build --release
cargo build -p y2m
cargo build -p y2m-server
```

### 6.2 运行标准

当前运行方式如下：

1. `y2m-server`：默认监听 `127.0.0.1:8080`
2. 服务端地址可由 `Y2M_SERVER_ADDR` 覆盖
3. 客户端通过 `y2m init / run / send / chat` 使用
4. 本地常驻服务进程应优先用 `tmux` 管理

### 6.3 测试标准

Rust 测试命令从 `y2m-rs/` 执行：

```bash
cargo check --workspace
cargo test --workspace
cargo test -p y2m
cargo test --test file_transfer_v3
cargo test --test cli_process_e2e -- --nocapture
```

测试分类标准：

1. 优先使用进程内集成测试验证协议、路由、文件传输与运行时行为
2. 只有在需要覆盖 CLI 解析、stdin 交互、真实子进程重连行为时，才补 CLI 进程 e2e
3. Windows 上部分 CLI e2e 可因子进程稳定性与编码问题被有条件忽略，但不应影响协议层覆盖

## 7. 工程约束标准

以下约束直接继承 [`agent.md`](../agent.md)，对人工开发与自动化工具同时生效：

1. 单个函数最长 50 行
2. 单个源码文件最长 500 行
3. 完成一个连贯任务后，必须提交 Git
4. Git 提交信息应完整描述“改了什么、为什么改”
5. 本地常驻开发进程统一用 `tmux` 管理
6. 前端项目统一使用 `pnpm`

补充约束：

1. 人类可读文档允许继续使用简体中文
2. 代码标识符、提交信息、公开 API 名保持英文
3. 前端目录当前按以下约定执行：
   - 管理端：`frontend-monorepo/apps/y2-manage`
   - 普通用户交互已改为 CLI-only（`y2m` 二进制），不再维护用户端 Web 应用

## 8. 文档维护标准

### 8.1 更新顺序

实现发生变化时，按以下顺序更新文档：

1. 先更新 [`当前实现说明.md`](./当前实现说明.md)
2. 再更新 [`工作进度.md`](./工作进度.md)
3. 若涉及导航、仓库说明、工程约束变化，再更新：
   - [`agent.md`](../agent.md)
   - 相关需求、设计、API、运维文档

### 8.2 文档职责分工

- `当前实现说明.md`：记录“已经实现了什么”
- `工作进度.md`：记录“当前做到了哪一步、下一步做什么”
- **`agent.md`**：记录“代码和协作约束”“仓库怎么找、命令怎么跑、测试怎么分层”
- 本文档：记录“当这些文档一起使用时，统一按什么标准执行”

### 8.3 冲突处理

若发现文档冲突：

1. 先按本文第 3 节优先级解释
2. 若冲突来自过期导航文档，应及时修正该文档
3. 不允许继续让过期文档长期作为事实来源

## 9. 当前推荐执行方式

针对当前仓库，开发时应按以下方式执行：

1. 先看 [`当前实现说明.md`](./当前实现说明.md)
2. 再看 [`工作进度.md`](./工作进度.md) 确认优先级
3. 写代码、查目录、构建命令、测试入口时均遵守并参考 [`agent.md`](../agent.md)
4. 若发现 **`agent.md`** 与实现现状冲突，立即以 `当前实现说明.md` 为准
