# 任务清单 v2 — 多 Agent 协作方向

> 基于 y2m-rs 基础设施，向多 Agent 协作编排演进。
> 用户端 Web（y2-chat）已移除，交互为 CLI-only。

---

## 模块一：传输层稳定（y2m-rs 债务清理）

> 目标：把现有消息总线的稳定性缺口补完，让后续 auth 和编排建立在可靠地基上。

| # | 任务 | 状态 | 依赖 |
|---|------|------|------|
| TR-01 | 断线重连形式化：明确会话恢复策略（重连后是否复用原 connection_id） | ⬜ | — |
| TR-02 | 断线重连：文件传输中断后与服务端传输态严格对齐（清理僵尸 transfer） | ⬜ | TR-01 |
| TR-03 | `command` 超时提示：可配置默认超时，超时后向发送方返回 `command_result`（exitCode=-1） | ⬜ | — |
| TR-04 | `command` 广播场景收齐策略（可选）：等待所有接收方返回或超时聚合 | ⬜ | TR-03 |
| TR-05 | chunk/header 异常边界：更多字段异常组合（非法 `frameType`、version 不匹配等） | ⬜ | — |
| TR-06 | Windows CLI e2e 稳定性：恢复或替换 `cli_process_e2e` 中 `cfg(windows)` ignore 的用例 | ⬜ | — |
| TR-07 | 服务端指标：接入 `metrics` crate，暴露连接数、路由量、传输量、心跳超时数 | ⬜ | — |
| TR-08 | `ConsolePlugin` 拆分：拆为独立 `message` / `json` / `command` / `file` 插件 crate | ⬜ | — |

## 模块二：统一认证中心（auth-service）

> 目标：独立认证服务，作为唯一身份源与权限源。Rust + Axum + SQLx + PostgreSQL + Redis。
> 设计文档：`../auth/统一认证中心详细设计-v1.md`、`../auth/统一认证中心API定义-v1.md`。

| # | 任务 | 状态 | 依赖 |
|---|------|------|------|
| AU-01 | 数据模型落地：users / groups / roles / permissions / role_permissions / user_group_roles | ⬜ | — |
| AU-02 | 数据模型落地：devices / sessions / audit_logs | ⬜ | AU-01 |
| AU-03 | JWT 基础设施：RS256 密钥对生成、签发、验证；JTI 唯一标识 | ⬜ | — |
| AU-04 | Web 登录：`POST /auth/web/login`，返回 Access Token + Refresh Token（HttpOnly Cookie） | ⬜ | AU-01, AU-03 |
| AU-05 | Web 刷新：`POST /auth/web/refresh`，轮换 Refresh Token，旧 Token 立即失效 | ⬜ | AU-04 |
| AU-06 | Web 登出：`POST /auth/web/logout`，销毁会话，清除 Cookie | ⬜ | AU-04 |
| AU-07 | CLI 登录：`POST /auth/cli/login`，接收 MAC + IP + osLoginUser + deviceNonce + deviceSignature | ⬜ | AU-02, AU-03 |
| AU-08 | CLI 首登审核流：新设备进入 `pending_review`，管理员审批后签发 trust token | ⬜ | AU-07 |
| AU-09 | 设备信任状态机：`unknown → trusted → suspended → revoked`，自动降级策略 | ⬜ | AU-08 |
| AU-10 | 权限决策：`GET /auth/me` 返回角色、权限、组、会话状态 | ⬜ | AU-01 |
| AU-11 | 服务间 introspect：`POST /auth/introspect`，供 y2m-server 调用校验 token | ⬜ | AU-03, AU-10 |
| AU-12 | Admin API：用户创建/禁用、角色权限分配、设备审核、审计日志查询 | ⬜ | AU-01~AU-11 |
| AU-13 | 审计日志：只追加写入，记录登录/鉴权/权限变更/设备状态变更 | ⬜ | AU-02 |
| AU-14 | IP 白名单：CIDR 规则匹配，连接时校验 | ⬜ | AU-07 |
| AU-15 | Redis 集成：短期会话态、token 吊销黑名单、限流计数 | ⬜ | AU-04 |

## 模块五：Agent CLI 包装器

> 目标：把 `y2m` CLI 扩展为长期运行的 Agent 进程，能接收任务、调用本地 AI CLI、回传结果。

| # | 任务 | 状态 | 依赖 |
|---|------|------|------|
| AG-01 | 新增 `y2m agent` 子命令：长期运行模式，连接后循环监听 Event | ⬜ | AC-05 |
| AG-02 | Agent 身份注册：启动时通过 `init` 注册，携带 `capabilities`（如 `react`、`typescript`、`architecture`） | ⬜ | AG-01 |
| AG-03 | 任务事件过滤：监听 `EventType::Json`，筛选 `messageType = task:*` 的消息 | ⬜ | AG-01 |
| AG-04 | 上下文文件生成：收到任务后，生成带上下文的任务文件（`context/<taskId>.md`） | ⬜ | AG-03 |
| AG-05 | 本地 CLI 调用封装：`spawn` 本地 AI CLI（claude / kimi / opencode / cursor / codex）并收集输出 | ⬜ | AG-04 |
| AG-06 | 任务完成回传：执行完毕后发送 `task:complete`，附带输出摘要和文件路径 | ⬜ | AG-05 |
| AG-07 | 修复任务上下文注入：收到 `task:reject-fix` 时，生成带历史记录 + 问题列表 + 上一版代码的修复上下文 | ⬜ | AG-06 |
| AG-08 | `ContextManager` 内存管理：只保留最近 20 个任务历史，项目规范常驻，定期持久化到 `/tmp/agent-context/` | ⬜ | AG-01 |

## 模块六：agent-orchestrator 编排服务

> 目标：任务调度、状态机管理、共享层锁定。可作为独立进程或 y2m-server 的 crate 插件。

| # | 任务 | 状态 | 依赖 |
|---|------|------|------|
| OR-01 | TaskStore：内存任务状态机（PENDING → ASSIGNED → RUNNING → COMPLETED → VERIFYING → NEEDS_FIX / APPROVED） | ⬜ | AC-02 |
| OR-02 | 迭代机制：验收不通过时 version+1，原 Agent 修复，历史上下文注入 | ⬜ | OR-01 |
| OR-03 | 需求变更级联：分析影响范围，触发相关任务迭代 | ⬜ | OR-02 |
| OR-04 | AgentRegistry：能力矩阵注册（capability → agent 列表）+ 负载跟踪 | ⬜ | AG-02 |
| OR-05 | Scheduler 评分算法：`capabilityMatch*0.4 + load*0.2 + successRate*0.2 + complexityFit*0.2` | ⬜ | OR-04 |
| OR-06 | 拓扑排序调度：按依赖关系排序，无依赖的先执行，有依赖的等待完成后启动 | ⬜ | OR-05 |
| OR-07 | SharedLayerManager：共享层预实现 → 锁定（checksum）→ 分发约束 → 验收时校验完整性 | ⬜ | AC-03 |
| OR-08 | 路由角色：Claude 作为 Router + Merger，Kimi 作为 Parser，OpenCode/Cursor 作为 Coder | ⬜ | OR-06 |
| OR-09 | 编排服务对接 y2m-server：通过 WebSocket 收发 `agent-collab` 消息 | ⬜ | SV-07 |
