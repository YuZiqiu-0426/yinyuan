## 2. 当前项目结构

当前已经落地的项目结构如下：

```text
y2m-rs/
  Cargo.toml
  src/
    main.rs
    line_editor.rs   ← reedline 行编辑器（chat 模式）
    printer.rs       ← 全局输出路由（chat/非 chat 模式统一接口）
    state/
      command_results.rs  ← command_result 按 request_id 聚合与防抖打印
  crates/
    common/
    server/
    client-core/
      sender_envelope.rs  ← 出站 metadata：senderIp / MAC / User / Os
  tests/
    support/
    text_e2e.rs
    json_e2e.rs
    command_e2e.rs
    file_transfer_v3.rs
    cli_*_e2e.rs   ← 真实子进程 CLI（Windows 上部分用例 cfg ignore，见 §3.9）
```

其中：

1. `crates/common`：协议模型、公共枚举、协议常量、二进制分片头、跨平台命令辅助方法
2. `crates/server`：服务端 `init / router / ws / session / transfer` 主链路
3. `crates/client-core`：客户端配置、连接、协议解析、运行时、插件注册、命令包构造；`sender_envelope` 为出站事件附加本机 IP/MAC/用户/OS
4. `src/main.rs`：`y2m` CLI 入口，负责 `init / run / send / chat`
5. `src/line_editor.rs`：reedline 行编辑器，仅 `chat` 模式启用，提供 `/` 命令下拉补全
6. `src/printer.rs`：全局输出路由；`chat` 模式安装 `ExternalPrinter`，其他模式回退 `println!`
7. `tests/`：当前 workspace 的端到端测试与测试辅助模块
8. 仓库根另有 **`frontend-monorepo/`**（前端 monorepo，仅含管理端 `y2-manage`）、**`auth-service/`**（统一认证 Axum **架子**，独立 `cargo`）、**`agent.md`** 等；前端与多 crate 的 `cargo` 约定以 **`agent.md`** 为准。

6. 文件传输 `v3`

## 3.4 CLI

当前 `y2m` CLI 已经完成以下命令：

1. `y2m init`
   - 保存本地配置
   - 支持 `--server-url`
   - 支持 `--group`
   - 支持 `--client`
   - 支持 `--token`
   - 支持 `--heartbeat-interval`
   - 支持 `--download-dir`
2. `y2m run`
   - 连接服务器
   - 打印控制台消息
   - 自动心跳
   - 支持 `--reconnect-interval-sec` 最小自动重连
   - 断线后自动重新 `init`
   - 收到 `command` 时执行本地命令并回传 `command_result`
   - 支持文件确认控制命令
3. `y2m send text`
4. `y2m send json`
5. `y2m send command`
   - **无 `--to` 时**可与服务端路由一致走**组内广播**（打印目标可为 `*`）；指定 `--to` 时为单播
   - 支持等待 `command_result`（收到首条结果后结束等待；与 `command_result` 聚合打印逻辑配合，见 §3.4 第 9 点）
   - 等待超时秒数：**`--timeout`** 优先，否则读配置 **`commandWaitTimeoutSec`**（可用 **`y2m init --command-wait-timeout`** 写入），再否则默认 **30**；与事件 metadata **`timeoutSec`** 及本地等待一致
6. `y2m send file`
   - **无 `--to` 时**可走组内广播；指定 `--to` 时为单播
   - 发送 `file_offer`
   - 等待 `file_accept / file_reject`
   - 发送二进制 chunk
   - 发送 `file_complete`
   - 等待最终 `ack`
   - 已有发送进度和阶段日志
7. `y2m chat`
   - 支持进入默认会话
   - 支持 `/to`
   - 支持 `/group`
   - 支持 `/json`
   - 支持 `/command`
   - 支持 `/file`
   - 支持 `/files`
   - 支持 `/accept`
   - 支持 `/reject`
   - 支持 `/abort`
   - 支持 `/status`
   - 支持 `/help`
   - 支持 `/exit`
   - 支持 `--reconnect-interval-sec` 最小自动重连
   - 断线后自动重新 `init`
   - 收到 `command` 时执行本地命令并回传 `command_result`
   - **输入行编辑器已升级为 reedline**：输入 `/` 自动弹出命令下拉菜单，支持补全与描述提示；服务端推送消息通过 `ExternalPrinter` 正确打印在提示符上方，不破坏正在编辑的输入行
8. **对端展示发送端终端信息**（依赖 `metadata` 中的 sender 四键；旧客户端无此字段则显示与以前一致）
   - 文本 / JSON：`[group][from | ip=… mac=… user=… os=…] …`
   - 收到 `command`：额外一行 `发送方终端: …`
   - 收到 `file_offer`：在「收到文件请求」后一行 `发送方终端: …`
   - `command_result` 汇总打印：每个客户端块下可有 `执行方: …`（与结果聚合逻辑配合，见 §3.10）
9. **`command_result` 按 `request_id` 聚合打印**
   - 同一 `request_id` 下多条结果先入缓冲，**约 250ms** 无新结果则按 **用户名字典序**（`BTreeMap`）成批打印；`y2m send command` 在等待结束时会 **立刻 flush**，避免进程退出丢失输出

## 3.5 文件传输当前状态

当前文件传输链路已经可以跑通：

1. 发送端支持 `send file` 和 `chat /file`
2. 接收端 **`file_offer` 接受策略**
   - 若配置中**显式设置了** `download_dir`（`y2m init --download-dir` 或 JSON 含 `downloadDir`）：收到 `file_offer` 后**自动** `move_to_incoming` 并发送 `file_accept`（无需 `/accept`）
   - 若配置中**未设置** `download_dir`：仍使用默认保存目录 `当前工作目录/downloads/`，但**不会**自动接受，需 `/accept` 或 `/reject`
3. 接收端支持：
   - `/files` 查看待确认和接收中的文件
   - `/accept <fileId>` 接收
   - `/reject <fileId>` 拒绝
   - `/abort <fileId>` 取消待确认或接收中的文件
4. 发送端支持：
   - `send file` 的阻塞式发送
   - `chat /file` 的后台发送
   - `chat /abort <fileId>` 取消外发文件
5. 接收文件时会：
   - 自动选择保存路径
   - 避免重名覆盖
   - 按配置中的 `download_dir` 或默认 `downloads/` 保存
   - 校验 `fileSize`
   - 校验 `sha256`
   - 校验通过后落盘
   - 返回 `ack`
6. 当前发送端和接收端都已经带有进度/阶段日志
7. `file_abort` 当前已完成：
   - 协议包构造
   - 服务端转发和传输状态清理
   - 发送侧收到 `file_abort` 后停止继续发送
   - 接收侧收到 `file_abort` 后清理本地接收状态
8. 已补的错误处理：
   - 未 `accept` 前发送 binary chunk 会被拒绝，并向发送方返回错误包
   - 非法 binary chunk 会向发送方返回 `invalid_message`
   - 缺失传输状态时的 `file_complete` 不会继续转发
   - `file_offer` 超过服务端 `max_file_size / max_chunk_size` 会直接被拒绝
   - `file_complete` 收到 `ack=rejected` 后会清理服务端传输状态
9. 当前最小重连一致性处理：
   - `run/chat` 断线后会自动重新 `init`
   - 重连前会清理待确认/接收中/外发中的文件状态
   - 重连成功后会本地回放未完成文件已失败，并提示“等待对方在线后重新发送”
10. 本地文件状态结构当前已进一步收敛：
   - `ConsoleState` 底层已改为单个 `LocalFileStore` 承载待确认 / 接收中 / 外发中三类状态
   - `LocalFileStore` 内部已进一步收敛为 `fileId -> LocalFileTransfer` 单映射，每个文件只对应一个本地状态对象
   - 原有状态读写 helper 已切到统一存储入口，减少多把锁分散访问
   - `LocalFileTransfer` 已开始采用共享字段 + `LocalFileState` 显式阶段表达，本地外发阶段不再额外维护独立枚举
   - `LocalFileTransfer` 现已内聚常见状态谓词与阶段迁移入口，helper 不再分散维护外发阶段判断
   - `move_to_incoming()` / `transition_to()` 已从布尔返回值改为结构化迁移错误，非法迁移原因可以被单独建模

## 3.6 chat 行编辑器（reedline）

`y2m chat` 已从 `BufRead::lines()` 行缓冲读取升级为基于 reedline 的行编辑器：

1. **命令下拉补全**
   - 输入 `/` 时自动触发 `ColumnarMenu`，列出所有 `/` 命令及其参数说明
   - 继续输入字符会实时过滤候选项；Tab 键循环选中，回车补全
   - 仅对第一个 token 补全（以 `/` 开头且不含空格），普通文本消息不触发
2. **输出与编辑行不冲突**
   - 所有运行时输出（收到消息、文件进度、重连提示等）统一经由 `printer::cprintln!` 宏路由
   - `chat` 模式启动时安装 `ExternalPrinter`，消息异步打印在光标行上方，不破坏当前输入内容
   - `run / send / init` 模式不安装 printer，`cprintln!` 自动回退到 `println!`，行为与之前完全一致
3. **输入信号处理**
   - `Ctrl+C`：清除当前输入行，继续等待（不退出）
   - `Ctrl+D`：触发 EOF，等同于 `/exit`
4. **关键模块**
   - `src/line_editor.rs`：`SlashCompleter` + `spawn_line_editor(tx) -> ExternalPrinter<String>`
   - `src/printer.rs`：`static PRINTER: OnceLock<ExternalPrinter<String>>` + `cprintln!` 宏

## 3.7 command 事件执行（Windows pwsh）

在 Windows 上，`command` 事件执行逻辑（`state/command.rs::execute_command`）已从 `cmd /C` 改为 `pwsh`：

1. 使用 `pwsh -NoProfile -NonInteractive -Command "[Console]::OutputEncoding = [System.Text.Encoding]::UTF8; <cmd>"`
2. 前置 `[Console]::OutputEncoding` 设置确保中文输出不乱码（`cmd` 在中文 Windows 输出 GBK，`from_utf8_lossy` 处理会得到乱码）
3. Linux/macOS 保持原有 `sh -c` 路径，不受影响
4. **前提**：目标机器需安装 PowerShell Core（`pwsh.exe`）；未安装时命令执行会返回"找不到程序"错误

注意：`y2m-common` 的 `default_shell_program() / default_shell_arg()` 保持不变（仍返回 `cmd /C`），该变更仅在 `command` 事件执行层生效，不影响协议层约定。

## 3.8 日志落文件

`y2m` 和 `y2m-server` 均已接入 `tracing-appender`，运行时自动将日志追加写入文件：

| 程序 | 默认日志文件 | 路径覆盖环境变量 |
|------|------------|----------------|
| `y2m` | `y2m.log`（当前目录） | `Y2M_LOG` |
| `y2m-server` | `y2m-server.log`（当前目录） | `Y2M_SERVER_LOG` |

1. 每次启动**追加**写入，不覆盖历史日志
2. 文件格式：带时间戳的纯文本（无 ANSI 色码）
3. 服务端额外保留 stderr 的 JSON 格式输出（方便日志采集系统对接）
4. 默认日志级别 `INFO`，通过 `RUST_LOG` 环境变量可调（如 `RUST_LOG=debug`）
5. 使用非阻塞 writer（`tracing_appender::non_blocking`），日志写入不阻塞 async 运行时

   - 真实 `chat` 进程下的多 chunk 乱序分片会立刻失败、本地清理并向发送方回 `file_abort`
   - 真实 `chat` 进程下的重复 chunk 会立刻失败、本地清理并向发送方回 `file_abort`
   - 真实 `chat` 进程下的 `totalChunks` 不一致会立刻失败、本地清理并向发送方回 `file_abort`

7. `src/main.rs` 单元测试
   - 非法 `move_to_incoming()` 会返回显式 `UnexpectedState`
   - 非法 `transition_to()` 会返回显式 `InvalidTransition`
   - 合法 `move_to_incoming()` 会重置进度并切到 `Receiving`
8. `y2m-client-core`：`command_results` 聚合与 **macOS** 下 `ifconfig` 解析 MAC 的单元测试（按 `cfg` 编译）
9. **Windows 上部分 CLI 进程 e2e**（`cli_*_e2e.rs` 等）带有 `cfg_attr(windows, ignore = "...")`：在 Windows 自动化环境里子进程输出/编码不稳定时跳过；协议与行为仍由进程内测试（如 `file_transfer_v3`）覆盖

## 3.12 管理端 `y2-manage`（`frontend-monorepo/apps/y2-manage`）

与 Rust 主仓并行维护的 **Angular 管理端**最小可用形态（Phase 0～2 范围），便于在无 `auth-service` 时继续壳与表单联调：
