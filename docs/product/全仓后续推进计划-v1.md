## 3. 总体策略：主线、副线、穿插

### 3.1 主线（建议优先投入）

**把 `auth-service` 从「仅架子」推进到「可与 `y2-manage` 联调的最小 HTTP API」。**

- **原因**：管理端已按 [`docs/auth/统一认证中心API定义-v1.md`](../auth/统一认证中心API定义-v1.md) 做 Mock；**下一阶段瓶颈**是真实或 **stub** 的 **`/api/v1/auth/web/*`**（至少与前端 `AUTH_API_PREFIX` 一致），否则生产形态无法验证。
- **与 relay 的关系**：`y2m-server` 在 `init` 阶段对 token 的校验（任务清单中的 **SV-01**、**AU-04** 等）依赖统一认证侧契约稳定；先做 **可预测的 HTTP 契约**（哪怕内存用户），有利于 relay 与双端并行开发。
- **数据库时机**：不必与「第一条路由」强绑定；可先 **内存 stub**（固定 JSON 信封、`OK` / 错误码与文档一致），再引入 PostgreSQL + SQLx（见 [`docs/auth/统一认证中心详细设计-v1.md`](../auth/统一认证中心详细设计-v1.md) §3、[`数据库设计与迁移方案-v1.md`](../auth/数据库设计与迁移方案-v1.md)）。

### 3.2 副线（可与主线并行）

**`y2-manage` Phase 3（用户列表等）在 Mock 或只读假数据下推进 UI。**

- **原因**：对 **AU-12**（用户管理 API）依赖较弱时可先做表格、分页占位、空/错状态、路由与权限占位，减少前端空闲等待。
- **衔接**：API 路径与类型先与 [`统一认证中心API定义-v1.md`](../auth/统一认证中心API定义-v1.md) / Admin 相关规划对齐；待 `auth-service` 或 BFF 就绪后切换 `useAuthMock` 与 `apiBaseUrl`。

### 3.3 穿插（长期工程、不阻塞产品闭环）

**`y2m-rs` 的 P0/P1 加固**（更细 chunk/重连形式化、配置扩展、`send file` 超时与配置对齐等）。

- **原因**：提升协议与 relay 可信度，但不阻塞「认证 + 管理端」主线。
- **参考**：《当前实现说明》§4、§7；《工作进度》§4、§5。

## 4. 分阶段推进建议（可独立验收）

### 阶段 A：`auth-service` 最小 HTTP（stub）

**目标**：`y2-manage` 在关闭 `useAuthMock`、指向本机时可完成「登录 + refresh + MFA（可先固定分支）」的**联调闭环**（不要求真实持久化）。

**建议交付物**：

1. 路由前缀 **`/api/v1`** 与文档一致；实现或 stub：**`POST /auth/web/login`**、**`POST /auth/web/refresh`**、（按需）**`POST /auth/web/mfa/verify`**，响应信封与 [`统一认证中心API定义-v1.md`](../auth/统一认证中心API定义-v1.md) 一致。
2. 错误码与 HTTP 状态策略与 [`错误码与异常语义规范-v1.md`](../auth/错误码与异常语义规范-v1.md) / API 定义对齐（可先子集）。
3. **CORS**：允许管理端开发源（`y2-manage` 的 `ng serve` 源）与 **credentials** 策略在设计允许范围内最小开放，便于 Cookie/CSRF 后续接入。
4. README 中写明：默认监听端口、与 `y2-manage` `environment.apiBaseUrl` 的对齐示例。

**不强制**：PostgreSQL、Redis、JWT 签名、真实 Argon2（可后续阶段 B/C）。

### 阶段 B：`auth-service` 持久化与真实登录

**目标**：用户与密码哈希落库、refresh 与 CSRF 策略按设计落地。

**建议交付物**：

1. PostgreSQL + SQLx（或设计文档选定栈）、迁移目录与最小 `users`/`sessions` 等表（以 [`数据库设计与迁移方案-v1.md`](../auth/数据库设计与迁移方案-v1.md) 为准）。
2. Redis（若设计需要）：吊销列表、限流或短期会话（见详细设计 §2.1）。
3. JWT 签发与校验、与 [`配置与密钥管理规范-v1.md`](../auth/配置与密钥管理规范-v1.md) 一致的秘密管理。

### 阶段 C：`y2m-server` 鉴权接入

**目标**：`init` 时校验 `InitPayload.token`（**introspect** 或等价），未通过则拒绝 `init`。

**建议交付物**：

1. `ServerConfig` 增加 **`auth_service_url`**（或等价配置），见任务清单 **SV-01**。
2. 与 `auth-service` 的联调说明写入 [`当前实现说明.md`](./当前实现说明.md) 与 [`y2m-rs/docs/quickstart.md`](../../y2m-rs/docs/quickstart.md)（若影响本地起服流程）。

### 阶段 D：`y2-manage` 业务模块与运维

**目标**：用户/角色/设备/审计等 MW 模块在真实或 staging API 上闭环；CI 与部署与 `auth-service`、relay 编排一致。

**衔接**：[`y2-manage-框架搭建任务表.md`](./y2-manage-框架搭建任务表.md)、[`任务清单-v2.md`](./任务清单-v2.md) 模块七。

## 5. 显式降优先或延后

以下项**不建议**作为当前全仓「第一条主线」，除非安全或合规强制要求提前：

1. **`y2m-rs` 大规模结构重构**（如 `ConsolePlugin` 拆多 crate）：工程收益高但对产品演示闭环帮助慢，适合穿插。
2. **文件断点续传级协议**：工作量大，依赖产品结论，见《当前实现说明》§4 / §7.3。
3. **Git 历史瘦身**：与功能交付无直接依赖，单列运维窗口处理。
