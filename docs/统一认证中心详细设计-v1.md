# 统一认证中心详细设计 v1

## 1. 设计目标

本设计文档用于指导统一认证中心（Auth Center）第一阶段实现，确保以下目标可落地：

- 同时支持 CLI 与 Web 的统一认证与鉴权。
- 角色体系支持 `super_admin`、`group_admin`、`user`。
- 权限覆盖 `text/json/command/file` 的发送和接收。
- 先上线文本聊天链路，但权限模型一次性设计完整。
- 与现有 Rust 服务（`y2m-server`）低耦合集成。

## 2. 总体架构

## 2.1 系统组件

- `auth-service`（Rust + Axum）：认证、会话、权限决策、设备信任、审计。
- `chat-service`（现有 y2m-server）：消息路由与业务执行，接入鉴权中间件。
- `user-web`（Next.js）：普通用户聊天与个人会话操作。
- `admin-web`（Angular）：用户、组、角色、权限、设备、审计管理。
- `postgres`：主数据存储（用户、角色、权限、会话、审计）。
- `redis`：短期会话态、token 吊销黑名单、限流计数。

### 2.2 边界与职责

- Auth Center 是唯一身份源与权限源。
- Chat Service 不保存账号密码，不做角色配置，只校验与执行。
- 两个 Web 前端都通过 Auth API 获取身份与权限，不直连数据库。

## 3. 技术选型

- 语言：Rust（与现有项目统一）
- Web 框架：Axum
- 数据访问：SQLx
- 数据库：PostgreSQL
- 缓存：Redis
- Token：JWT（建议 RS256）
- 密码哈希：Argon2id
- 日志与追踪：tracing
- 配置：dotenvy + 环境变量

## 4. 模块设计

## 4.1 auth-service 内部模块

- `api`：HTTP 路由与请求校验。
- `application`：认证编排、权限决策、会话管理。
- `domain`：核心实体与业务规则（User、Role、Permission、Device、Session）。
- `infra`：Postgres/Redis/JWT/密码哈希实现。
- `audit`：审计记录入口与异步写入。

### 4.2 鉴权中间件（供 chat-service 复用）

- `verify_access_token`：校验 JWT 签名与标准字段。
- `verify_ip_binding`：校验请求源 IP 是否满足策略。
- `load_permission_snapshot`：加载 token 声明或缓存中的权限快照。
- `authorize(action)`：基于动作进行原子权限判定。

## 5. 数据模型设计

以下为逻辑模型，物理字段按 PostgreSQL 实现。

### 5.1 用户与组织模型

- `users`
  - `id` (uuid, pk)
  - `username` (varchar, unique)
  - `password_hash` (varchar)
  - `status` (active/disabled/locked)
  - `created_at`, `updated_at`

- `groups`
  - `id` (uuid, pk)
  - `name` (varchar, unique)
  - `status` (active/disabled)
  - `created_at`, `updated_at`

- `user_group_roles`
  - `id` (uuid, pk)
  - `user_id` (fk users.id)
  - `group_id` (fk groups.id)
  - `role_id` (fk roles.id)
  - unique(`user_id`,`group_id`,`role_id`)

### 5.2 角色与权限模型

- `roles`
  - `id` (uuid, pk)
  - `code` (`super_admin` / `group_admin` / `user`)
  - `name` (varchar)
  - `scope` (global/group)
  - `is_system` (bool)

- `permissions`
  - `id` (uuid, pk)
  - `code` (如 `text.send`)
  - `description` (varchar)

- `role_permissions`
  - `id` (uuid, pk)
  - `role_id` (fk roles.id)
  - `permission_id` (fk permissions.id)
  - unique(`role_id`,`permission_id`)

### 5.3 设备与会话模型

- `devices`
  - `id` (uuid, pk)
  - `user_id` (fk users.id)
  - `device_fingerprint` (varchar, unique)
  - `mac_address` (varchar)
  - `ip_address` (inet)
  - `os_login_user` (varchar)
  - `status` (unknown/trusted/suspended/revoked)
  - `last_seen_at`
  - `created_at`, `updated_at`

- `sessions`
  - `id` (uuid, pk)
  - `user_id` (fk users.id)
  - `client_type` (web/cli)
  - `device_id` (nullable fk devices.id)
  - `session_state` (active/suspended_readonly/revoked)
  - `access_jti` (varchar, unique)
  - `refresh_jti` (varchar, unique, nullable)
  - `bound_ip` (inet)
  - `expires_at`
  - `revoked_at` (nullable)
  - `created_at`

### 5.4 审计模型

- `audit_logs`
  - `id` (uuid, pk)
  - `actor_user_id` (nullable fk users.id)
  - `event_type` (varchar)
  - `target_type` (varchar)
  - `target_id` (varchar)
  - `result` (success/failed)
  - `reason` (text, nullable)
  - `ip` (inet)
  - `user_agent` (text)
  - `payload` (jsonb)
  - `created_at`

## 6. Token 与会话设计

### 6.1 Access Token Claims

- `iss`：auth-service
- `sub`：用户 ID
- `aud`：`web` 或 `cli` 或 `chat-service`
- `exp` / `iat`
- `jti`：token 唯一标识
- `role_codes`：角色列表
- `permission_codes`：原子权限列表
- `group_ids`：可访问组列表
- `ip_hash`：绑定 IP 的哈希值
- `device_id`：CLI 可选

### 6.2 Refresh Token 策略

- Refresh Token 仅用于 Web，保存在 HttpOnly Cookie。
- 采用轮换机制：每次刷新后旧 `refresh_jti` 立即失效。
- 检测到重放时，吊销该用户所有活跃会话并触发审计告警。
- CLI 不接入 Refresh Token 机制，也不接收 Refresh 失效通知。
- Refresh Token 失效后仅影响操作权限，不影响已建立广播监听连接。
- 当 Web Refresh Token 失效时，对应会话降级为 `suspended_readonly`：
  - 允许接收（`*.recv`）
  - 禁止操作（`*.send`）
  - 恢复为可操作状态需重新登录并建立新会话
- 若触发高风险事件（token 重放、管理员强制下线、异地异常登录），会话必须直接进入 `revoked` 并断链。

### 6.3 CLI 会话策略

- CLI 初次登录成功后签发短期 Access Token + 可选长期 Device Token。
- 每次会话都需重新提交并校验 `mac + ip + os_login_user`，不放宽关键因子校验。

## 7. 认证与鉴权流程

### 7.1 Web 登录流程

1. 用户提交用户名密码到 `/auth/web/login`。  
2. 认证中心校验账号状态与密码。  
3. 认证通过后创建 session，签发 Access + Refresh。  
4. 后续请求经中间件校验 JWT、IP、会话有效性。  
5. 权限不足返回 `AUTH_PERMISSION_DENIED`。  

### 7.2 CLI 登录流程

1. CLI 首次登录必须提交 `username + group_name + mac + ip + os_login_user` 到 `/auth/cli/login`。  
2. 认证中心计算设备指纹并校验设备状态。  
3. 首次登录进入待审核队列，需由 `group_admin` 或 `super_admin` 审核通过。  
4. 审核通过后创建 CLI session 并签发 token。  
5. chat-service 收到请求后校验 token 与 action 权限。  

### 7.4 每会话 CLI 设备因子校验

1. 每次 CLI 建立新会话时都重新上报 `mac + ip + os_login_user`。  
2. `ip` 以服务端连接源地址为准，不信任客户端上报。  
3. `mac + os_login_user + nonce` 需通过设备私钥签名校验。  
4. 认证中心将本次因子与设备绑定记录比对。  
5. 任一关键因子不匹配时拒绝会话并写入审计。  
6. 匹配通过后才允许进入消息收发流程。  

### 7.3 消息发送鉴权流程（chat-service）

1. 验签 access token。  
2. 校验 IP 绑定策略。  
3. 根据动作映射权限码（例如发送文本 -> `text.send`）。  
4. 校验通过再进入消息路由。  
5. 拒绝时写审计日志并返回标准错误码。  

### 7.5 会话状态机与运行时校验

- `active`：允许收发（前提是对应权限存在）。
- `suspended_readonly`：仅允许接收，禁止发送与管理操作。
- `revoked`：会话立即失效，拒绝所有请求并断开连接。
- 服务端必须在每次操作和每次广播分发前校验会话状态，不能只在握手时校验一次。

## 8. API 设计（v1）

## 8.1 Auth API（核心）

- `POST /auth/web/login`
- `POST /auth/web/refresh`
- `POST /auth/web/logout`
- `POST /auth/cli/login`
- `POST /auth/cli/logout`
- `GET /auth/me`（返回用户、角色、权限、组）
- `POST /auth/introspect`（服务间校验）

### 8.2 Admin API（管理端）

- `POST /admin/users`
- `PATCH /admin/users/{id}/status`
- `POST /admin/groups`
- `POST /admin/user-group-roles`
- `GET /admin/roles`
- `PUT /admin/roles/{id}/permissions`
- `GET /admin/devices`
- `PATCH /admin/devices/{id}/status`
- `GET /admin/audit-logs`

### 8.3 Chat 相关 API（授权入口）

- `POST /chat/text/send`
- `GET /chat/text/history`
- WebSocket 握手时携带 access token，连接建立前完成鉴权。
- `json/command/file` 在 v1 不开放前端 UI，仅保留后端鉴权与扩展接口。

## 9. 权限决策策略

### 9.1 动作映射表

- `chat.text.send` -> `text.send`
- `chat.text.recv` -> `text.recv`
- `chat.json.send` -> `json.send`
- `chat.json.recv` -> `json.recv`
- `chat.command.send` -> `command.send`
- `chat.command.recv` -> `command.recv`
- `chat.file.send` -> `file.send`
- `chat.file.recv` -> `file.recv`

### 9.2 决策规则

- `super_admin` 默认拥有全局权限，可跨组。
- `group_admin` 仅在所属组内具备管理与通信权限，仅能创建本组用户。
- `user` 仅具备显式授权权限。
- 拒绝默认优先：未显式允许即拒绝。
- 用户创建规则：`super_admin` 可创建任意分组用户，`group_admin` 仅可创建所属分组用户。

## 10. 安全设计

- 所有接口强制 HTTPS。
- 管理端在 v1 强制 MFA（`super_admin` 必选，`group_admin` 默认开启）。
- 登录限流（用户名 + IP 维度）。
- Web 刷新接口强制 CSRF 防护：`SameSite=Strict` + CSRF Token + Origin/Referer 校验。
- 密码强度策略（最小长度、复杂度、历史重用限制）。
- 高危操作二次确认（权限提升、禁用用户、吊销会话）。
- 审计日志只追加，不允许修改删除。
- IP 校验按 CIDR 白名单策略增强，支持单 IP、CIDR 网段与优先级规则。

## 11. 可观测性与运维

- 统一日志字段：`trace_id`、`user_id`、`ip`、`action`、`result`。
- 指标：登录成功率、鉴权拒绝率、token 刷新率、设备异常率。
- 告警：短时高频登录失败、token 重放、权限拒绝异常飙升。
- 密钥轮换：JWT 私钥定期轮换并保留公钥历史窗口。
- 权限变更与会话吊销全节点生效 SLA：`<= 5s`，超时触发告警。

## 12. 部署方案

### 12.1 环境划分

- `dev`：本地联调
- `staging`：预发布验收
- `prod`：生产

### 12.2 组件部署

- `auth-service` 与 `chat-service` 独立部署，统一通过 API 网关暴露。
- Postgres 主从或高可用实例，Redis 哨兵或托管版。
- 前端分别独立部署：Next（用户端）与 Angular（管理端）。

### 12.3 配置与密钥

- 所有密钥通过环境变量或密钥管理服务注入。
- 严禁将私钥、密码写入仓库。

## 13. 里程碑拆解

### Phase 1：认证中心可用

- 完成用户登录、CLI 登录、token 刷新、会话吊销。
- 完成角色权限表与基础管理 API。

### Phase 2：文本聊天鉴权闭环

- chat-service 接入中间件并完成 `text` 收发鉴权。
- 用户端 Next.js 完成登录与文本聊天页面。

### Phase 3：管理端可用

- Angular 完成用户管理、角色权限管理、审计查询。

## 14. 风险与应对

- MAC 地址在部分环境稳定性不足：增加设备指纹组合因子与人工确认兜底。
- IP 严格绑定会影响移动网络用户：支持风险分级策略（严格/宽松）。
- 权限模型复杂导致误配：提供默认角色模板和变更审计回滚能力。
