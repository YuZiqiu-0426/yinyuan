# 隐元消息系统需求文档

## 1. 文档目标

本文档定义隐元消息系统第一版的整体设计与实现约束，覆盖以下内容：

1. 服务端职责与实现边界
2. 客户端 `core + plugin` 架构
3. 统一消息协议
4. 文件传输 `v3` 方案
5. `init`、心跳、错误、日志规范
6. CLI 交互与插件接口约束

本文档的目标不是描述产品界面，而是提供一版可以直接指导 Rust 开发的实现规范。

## 2. 版本范围

第一版明确采用以下范围：

1. 服务端与客户端均由 Rust 实现
2. 传输层使用 `WebSocket`
3. 控制消息使用 `JSON` 文本帧
4. 文件传输使用 `v3` 方案：控制消息走 JSON，文件内容走二进制分片帧
5. 服务端只做在线消息中转，不做离线消息存储
6. 第一版不接数据库
7. 第一版不做复杂鉴权，但协议中预留 `token`
8. 插件采用静态注册，不做动态加载 `.dll/.so`

## 3. 总体架构

系统由两部分组成：

1. 服务端：负责连接管理、在线会话注册、消息路由、文件中转、心跳检查、错误返回、日志记录
2. 客户端：由 `core` 和多个 `plugin` 组成，`core` 负责连接和分发，`plugin` 负责处理不同业务消息

### 3.1 服务端职责

服务端本质上是一个在线消息中转站，但不是“只转发字节流”，还需要承担以下协议职责：

1. 处理连接建立与 `init`
2. 为客户端分配默认 `groupName` 和 `clientName`
3. 检测同组内 `clientName` 冲突
4. 维护在线会话表
5. 按 `groupName` 和 `clientName` 路由消息
6. 管理文件传输会话
7. 处理心跳和超时断开
8. 返回结构化错误
9. 输出结构化日志

### 3.2 客户端职责

客户端由 `core` 和 `plugin` 两层组成：

1. `core` 负责配置、连接、重连、心跳、命令转消息、消息分发、文件传输会话管理
2. `plugin` 负责处理 `text`、`command`、`json`、`file` 等业务消息

## 4. 服务端设计

### 4.1 技术角色

服务端建议使用以下技术栈：

1. `tokio`
2. `axum` 或 `tokio-tungstenite`
3. `serde`
4. `uuid`
5. `tracing`

### 4.2 在线会话规则

服务端内存中至少维护两张索引表：

1. `groupName -> clientName -> Session`
2. `connectionId -> SessionMeta`

`Session` 至少包含以下字段：

1. `connectionId`
2. `groupName`
3. `clientName`
4. `remoteAddr`
5. `connectedAt`
6. `lastHeartbeatAt`
7. `capabilities`

### 4.3 命名规则

客户端连接成功前必须经过 `init` 阶段，命名规则如下：

1. `groupName` 为空时，服务端默认使用 `default`
2. `clientName` 为空时，服务端生成 UUID 作为最终 `clientName`
3. 同一 `groupName` 下 `clientName` 不允许重复
4. 若同组重名，服务端拒绝新连接，返回 `duplicate_client_name`
5. `from` 不信任客户端填写，统一由服务端根据会话身份补齐

### 4.4 路由规则

服务端路由规则如下：

1. 指定 `target.groupName` 和 `target.clientName`：定向发送
2. 只指定 `target.groupName`：分组广播
3. 未指定 `target.groupName`：默认使用发送方所在分组
4. `text` 和 `json` 支持分组广播
5. `command` 第一版仅支持单播，不支持广播
6. `file` 第一版仅支持单播，不支持广播
7. 目标客户端不存在时，返回 `client_not_found`
8. 目标分组不存在时，返回 `group_not_found`

### 4.5 服务端内部模块建议

建议拆分为以下模块：

1. `listener`：监听 WebSocket 连接
2. `protocol`：负责 JSON 包解析、二进制分片解析、协议校验
3. `session_manager`：管理在线会话表
4. `router`：负责路由文本、命令、JSON、文件控制消息
5. `transfer_manager`：负责文件传输会话和二进制分片转发
6. `heartbeat`：处理心跳和超时清理
7. `error`：统一错误码与错误响应
8. `logging`：统一结构化日志

## 5. 客户端设计

### 5.1 core

`core` 由 Rust 实现，负责客户端基础能力。

启动后核心流程如下：

1. 读取本地配置
2. 建立 WebSocket 连接
3. 发送 `init`
4. 初始化 `PluginRegistry`
5. 注册各个 plugin
6. 启动心跳
7. 监听服务端消息
8. 将业务消息分发给对应 plugin

建议拆分以下模块：

1. `config`：本地配置读取和保存
2. `connection`：WebSocket 建连、收发、断线重连
3. `protocol`：协议编解码
4. `plugin_registry`：plugin 注册与分发
5. `command_bus`：CLI 命令转统一消息对象
6. `transfer_manager`：文件发送与接收会话
7. `session`：维护 chat 会话上下文

### 5.2 plugin

每个 plugin 是一个独立模块，第一版采用静态注册。

推荐结构：

```text
crates/
  common/
  server/
  client/
    core/
    plugins/
      message_plugin/
      command_plugin/
      file_plugin/
      json_plugin/
```

推荐接口：

```rust
use async_trait::async_trait;

#[async_trait]
pub trait Plugin: Send + Sync {
    fn name(&self) -> &'static str;
    fn supports(&self) -> &'static [EventType];
    async fn on_load(&mut self, ctx: PluginContext) -> anyhow::Result<()>;
    async fn on_event(&self, ctx: &PluginContext, event: &EventPacket) -> anyhow::Result<()>;
}
```

其中 `EventPacket` 表示已经完成 `kind = event` 解析后的统一业务事件对象。

`PluginRegistry` 职责：

1. 保存所有已注册 plugin
2. 建立 `EventType -> plugin 列表` 索引
3. 在启动时调用 `register()`
4. 在运行时调用 `dispatch()`
5. 某个 plugin 失败时记录日志，但不影响其他 plugin

## 6. 统一协议总览

### 6.1 传输层

协议统一基于 `WebSocket`：

1. 文本帧：承载 JSON 控制消息
2. 二进制帧：承载文件分片

### 6.2 通用 JSON 包结构

所有控制消息统一使用以下结构：

```json
{
  "version": "v3",
  "kind": "init",
  "requestId": "3f0a5a0d-8a76-4f40-bdfa-75ef8c0b0aa1",
  "timestamp": 1710000000,
  "source": {
    "groupName": "group1",
    "clientName": "alice"
  },
  "target": {
    "groupName": "group1",
    "clientName": "bob"
  },
  "payload": {}
}
```

字段说明：

1. `version`：协议版本，第一版固定为 `v3`
2. `kind`：控制消息类型
3. `requestId`：请求标识，用于日志、追踪和响应关联
4. `timestamp`：发送时间戳，Unix 秒或毫秒均可，但全系统必须统一
5. `source`：消息来源，服务端最终以在线会话身份为准
6. `target`：消息目标，用于单播或广播
7. `payload`：业务负载

### 6.3 kind 枚举

`kind` 定义如下：

1. `init`
2. `init_ack`
3. `heartbeat`
4. `heartbeat_ack`
5. `event`
6. `ack`
7. `error`

### 6.4 source 和 target 规则

规则如下：

1. `source` 在客户端发出时可以填写本地值，但服务端转发时必须重写为真实会话身份
2. `target.groupName` 为空时，默认使用发送方当前分组
3. `target.clientName` 为空时，表示对该分组广播
4. `init` 阶段的 `source` 和 `target` 可以为空
5. 服务端回包中的 `source.clientName` 可以固定写 `server`

## 7. 连接生命周期

### 7.1 init

客户端连接建立后发送 `init`：

```json
{
  "version": "v3",
  "kind": "init",
  "requestId": "req-init-001",
  "timestamp": 1710000000,
  "source": null,
  "target": null,
  "payload": {
    "groupName": "group1",
    "clientName": "dongming",
    "token": null,
    "capabilities": {
      "fileTransferV3": true,
      "commandPlugin": true,
      "jsonPlugin": true
    }
  }
}
```

### 7.2 init_ack

服务端成功响应：

```json
{
  "version": "v3",
  "kind": "init_ack",
  "requestId": "req-init-001",
  "timestamp": 1710000001,
  "source": {
    "groupName": "system",
    "clientName": "server"
  },
  "target": {
    "groupName": "group1",
    "clientName": "dongming"
  },
  "payload": {
    "connectionId": "conn-001",
    "groupName": "group1",
    "clientName": "dongming",
    "heartbeatIntervalSec": 60,
    "heartbeatTimeoutSec": 150,
    "maxFileSize": 104857600,
    "maxChunkSize": 262144
  }
}
```

### 7.3 init 阶段规则

1. 未收到 `init_ack` 前，客户端不得发送业务消息
2. `init` 校验失败时，服务端返回 `error`
3. `init_ack` 必须明确返回最终生效的 `groupName` 和 `clientName`
4. 客户端需要使用 `init_ack` 中返回的值更新本地运行态

### 7.4 心跳

客户端按固定周期发送 `heartbeat`：

```json
{
  "version": "v3",
  "kind": "heartbeat",
  "requestId": "hb-001",
  "timestamp": 1710000060,
  "source": {
    "groupName": "group1",
    "clientName": "dongming"
  },
  "target": {
    "groupName": "system",
    "clientName": "server"
  },
  "payload": {}
}
```

服务端返回 `heartbeat_ack`：

```json
{
  "version": "v3",
  "kind": "heartbeat_ack",
  "requestId": "hb-001",
  "timestamp": 1710000060,
  "source": {
    "groupName": "system",
    "clientName": "server"
  },
  "target": {
    "groupName": "group1",
    "clientName": "dongming"
  },
  "payload": {}
}
```

心跳规则：

1. 客户端默认每 60 秒发送一次
2. 服务端超过 150 秒未收到心跳则断开连接并清理会话
3. 客户端心跳失败后进入重连流程

## 8. event 消息格式

### 8.1 event 通用结构

业务消息统一放入 `kind = event` 的 `payload` 中：

```json
{
  "version": "v3",
  "kind": "event",
  "requestId": "req-1001",
  "timestamp": 1710000100,
  "source": {
    "groupName": "group1",
    "clientName": "alice"
  },
  "target": {
    "groupName": "group1",
    "clientName": "bob"
  },
  "payload": {
    "type": "text",
    "metadata": {}
  }
}
```

`payload.type` 定义如下：

1. `text`
2. `command`
3. `command_result`
4. `json`
5. `file_offer`
6. `file_accept`
7. `file_reject`
8. `file_complete`
9. `file_abort`

### 8.2 text

```json
{
  "type": "text",
  "content": "hello world",
  "metadata": {
    "contentType": "text/plain"
  }
}
```

说明：

1. 支持单播和分组广播
2. `content` 为 UTF-8 文本

### 8.3 command

```json
{
  "type": "command",
  "content": "ls -la",
  "metadata": {
    "timeoutSec": 30
  }
}
```

说明：

1. 第一版仅支持单播
2. 接收方是否执行由 `command plugin` 决定
3. 建议客户端默认关闭远程命令执行，需显式启用

### 8.4 command_result

```json
{
  "type": "command_result",
  "content": "",
  "metadata": {
    "requestId": "req-cmd-001",
    "exitCode": 0,
    "stdout": "hello\n",
    "stderr": "",
    "durationMs": 15
  }
}
```

### 8.5 json

```json
{
  "type": "json",
  "content": {
    "message": "hello"
  },
  "metadata": {
    "contentType": "application/json"
  }
}
```

说明：

1. 推荐 `content` 直接使用 JSON 对象
2. 支持单播和分组广播

## 9. 文件传输 v3

### 9.1 设计原则

文件传输直接采用 `v3`：

1. 文件控制消息走 JSON 文本帧
2. 文件内容走 WebSocket 二进制帧
3. 服务端不将文件完整读入内存，只维护传输会话并转发分片
4. 第一版文件传输只支持单播

### 9.2 文件传输流程

传输流程如下：

1. 发送方发送 `file_offer`
2. 接收方返回 `file_accept` 或 `file_reject`
3. 发送方开始发送二进制分片
4. 所有分片发送完成后，发送方发送 `file_complete`
5. 接收方校验成功后返回 `ack`
6. 任一步失败可发送 `file_abort`

### 9.3 file_offer

```json
{
  "version": "v3",
  "kind": "event",
  "requestId": "req-file-001",
  "timestamp": 1710000200,
  "source": {
    "groupName": "group1",
    "clientName": "alice"
  },
  "target": {
    "groupName": "group1",
    "clientName": "bob"
  },
  "payload": {
    "type": "file_offer",
    "content": null,
    "metadata": {
      "fileId": "file-001",
      "fileName": "hello.txt",
      "fileSize": 11,
      "contentType": "text/plain",
      "sha256": "64ec88ca00b268e5ba1a35678a1b5316d212f4f36631ecf0f9f0d5f7fdf6e8be",
      "chunkSize": 262144,
      "totalChunks": 1
    }
  }
}
```

### 9.4 file_accept

```json

  "target": {
    "groupName": "group1",

  },

    "content": null,
    "metadata": {
      "fileId": "file-001",

  }

```

### 9.5 file_reject

```json
{

    "groupName": "group1",
    "clientName": "bob"

    "clientName": "alice"

  "payload": {
    "type": "file_reject",
    "content": null,
    "metadata": {
      "fileId": "file-001",
      "reason": "file_too_large"
    }
  }
}
```

### 9.6 二进制分片帧

每个文件分片使用一个 WebSocket 二进制帧，帧格式如下：

1. `magic`：4 字节，固定为 `Y2MB`
2. `version`：1 字节，固定为 `3`
3. `frameType`：1 字节，固定为 `1`，表示 `file_chunk`
4. `reserved`：2 字节，保留
5. `fileId`：16 字节 UUID 二进制
6. `chunkIndex`：4 字节无符号整数，小端
7. `totalChunks`：4 字节无符号整数，小端
8. `payloadSize`：4 字节无符号整数，小端
9. `payload`：实际分片字节

约束如下：

1. `chunkSize` 默认 256 KB，可配置
2. 服务端只转发分片，不解码文件内容
3. WebSocket 基于 TCP，天然有序可靠，因此第一版不做分片级 ACK
4. 若连接断开或传输被取消，则本次文件传输失败

```json

  "source": {
    "groupName": "group1",

```

### 9.9 服务端文件传输职责

服务端对文件传输需要额外维护 `transfer registry`：

1. `fileId -> senderSession`
2. `fileId -> receiverSession`
3. `fileId -> status`
4. `fileId -> requestId`

服务端规则：

1. 收到 `file_offer` 后创建传输会话
2. 只有收到 `file_accept` 后，才允许转发该 `fileId` 的二进制分片
3. 若收到未知 `fileId` 的分片，直接丢弃并记错误日志
4. 收到 `file_complete` 或 `file_abort` 后清理传输会话

## 11. CLI 设计

### 11.1 命令入口

CLI 统一使用 `y2m`：

```bash
y2m init --host 127.0.0.1 --port 8080 --group group1 --client dongming
y2m run
y2m send text --to alice --group group1 "hello world"
y2m send command --to alice --group group1 -- "ls -la"
y2m send json --group group1 '{"message":"hello"}'
y2m send file --to alice --group group1 ./hello.txt
y2m chat --to alice --group group1
y2m chat --group group1
y2m chat
```

### 11.2 命令语义

1. `init`：初始化本地配置
2. `run`：启动常驻客户端，建立连接并持续监听
3. `send text`：发送文本消息
4. `send command`：发送命令执行请求
5. `send json`：发送 JSON 消息
6. `send file`：触发 `v3` 文件传输流程
7. `chat`：进入交互式会话

### 11.3 chat 会话规则

进入 chat 时参数规则：

1. `--to` 可省略
2. `--group` 可省略
3. `--to` 省略时，默认向当前分组广播
4. `--group` 省略时，默认使用本地当前分组
5. `y2m chat` 表示进入当前分组广播会话

会话内命令建议如下：

```bash
/to lucun
/group group2
/file ./hello.html
/json {"message":"hello"}
/command ls -la
/status
/help
/exit
```

会话规则：

1. 非 `/` 开头输入按文本消息处理
2. 当前会话未指定 `to` 时，文本消息默认广播到当前分组
3. 当前会话指定了 `to` 时，文本消息默认单播给目标用户
4. `/file` 走 `file_offer -> file_accept -> 二进制分片 -> file_complete`
5. `/json` 先做本地 JSON 校验
6. `/command` 第一版只允许在单播会话中使用

## 12. 客户端消息处理与插件职责

### 12.1 message plugin

负责处理 `payload.type = text`：

1. 打印接收到的文本消息
2. 将本地文本输入转换为 `event/text`
3. 支持广播和单播

### 12.2 command plugin

负责处理 `payload.type = command` 和 `command_result`：

1. 收到 `command` 后校验是否允许执行
2. 执行命令并捕获退出码、标准输出、标准错误
3. 将结果封装为 `command_result`
4. 建议默认关闭，需显式开启

### 12.3 json plugin

负责处理 `payload.type = json`：

1. 校验 JSON 合法性
2. 将结构化 JSON 分发给上层业务
3. 在 CLI 中以格式化方式输出

### 12.4 file plugin

负责处理 `file_offer`、`file_accept`、`file_reject`、`file_complete`、`file_abort` 和二进制分片：

1. 收到 `file_offer` 后决定接受或拒绝
2. 接收二进制分片并落盘
3. 在 `file_complete` 阶段校验文件大小和 `sha256`
4. 成功后返回 `ack`

## 13. 日志规范

### 13.1 日志原则

服务端和客户端都必须输出结构化日志，建议使用 JSON Lines。

1. `timestamp`
2. `level`
3. `side`，取值 `server` 或 `client`
4. `service`
5. `event`
6. `requestId`
7. `connectionId`
8. `groupName`
9. `clientName`
10. `targetGroupName`
11. `targetClientName`
12. `fileId`
13. `messageType`
14. `errorCode`
15. `durationMs`

1. `connection_opened`

## 14. Rust 公共模型建议

建议在 `common` crate 中定义以下模型：

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Endpoint {
    pub group_name: Option<String>,
    pub client_name: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum Kind {
    Init,
    InitAck,
    Heartbeat,
    HeartbeatAck,
    Event,
    Ack,
    Error,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, Hash)]
#[serde(rename_all = "snake_case")]
pub enum EventType {
    Text,
    Command,
    CommandResult,
    Json,
    FileOffer,
    FileAccept,
    FileReject,
    FileComplete,
    FileAbort,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EventPacket {
    pub event_type: EventType,
    pub content: serde_json::Value,
    pub metadata: serde_json::Value,
}
```

## 15. 实现顺序建议

推荐按以下顺序落地：

1. 先实现 `common` 协议模型
2. 再实现服务端 `init`、心跳、路由、错误返回
3. 实现客户端 `core`、`PluginRegistry`、`message plugin`
4. 实现 `json plugin`
5. 实现 `command plugin`
6. 最后实现 `file transfer v3`
